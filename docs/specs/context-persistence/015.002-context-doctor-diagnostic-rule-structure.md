# Execution Spec: 015.002-context-doctor-diagnostic-rule-structure

## Feature
- context-persistence

## Purpose
- Define a clean internal rule structure for feature-scoped context diagnostics so `execution_spec_drift` and future doctor rules fit the existing diagnostics model cleanly.
- Keep `context doctor` and `verify context` deterministic, composable, and easy to extend.
- Make new diagnostics easier to add without scattering ad hoc checks across commands and services.

## Scope
- Define the internal diagnostic-rule shape for context doctor.
- Define the minimum fields a rule must produce.
- Define how doctor and verify-context should consume those rule outputs.
- Refactor only as much as necessary to keep the new `execution_spec_drift` rule and future rules clean.
- Keep this focused on internal rule structure, not user-facing policy changes beyond what the rule model requires.

## Constraints
- Preserve current external command behavior except where needed for cleaner deterministic diagnostics.
- Keep JSON contracts stable unless additive changes are necessary.
- Keep diagnostics feature-scoped.
- Keep severity/proceed/repair semantics deterministic.
- Reuse existing doctor and verify-context infrastructure where practical.
- Do not redesign the entire context subsystem.
- Do not introduce plugin-style extensibility here unless it naturally fits the current architecture.

## Requested Changes

### 1. Introduce an Internal Context Diagnostic Rule Model

Create or formalize an internal rule model for feature-scoped context diagnostics.

A rule should encapsulate:
- a stable diagnostic code
- a deterministic evaluation method
- the conditions under which it emits a diagnostic
- structured details payload
- recommended actions
- severity or status classification compatible with doctor/verify-context

This may be implemented as a class, value object, interface, or similar internal abstraction, whichever fits the existing architecture most naturally.

### 2. Define Required Rule Output Shape

Each diagnostic rule must be able to produce a normalized output shape with at least:

- `code`
- `status` or equivalent severity
- `message`
- `details`
- `recommended_actions`

This normalized output must be what doctor aggregates and what verify-context interprets.

### 3. Separate Rule Evaluation from Presentation

Keep rule evaluation distinct from:
- CLI text rendering
- JSON rendering
- verify-context proceed/block summary logic

Rule evaluation should answer:
- whether a diagnostic exists
- what its structured payload is

Rendering and command-level summarization should consume that normalized output instead of embedding rule-specific conditionals.

### 4. Add an Execution-Spec Drift Rule Using the New Structure

Implement `execution_spec_drift` using the rule model introduced by this spec.

This rule should become the reference implementation for future feature-scoped doctor checks.

### 5. Define Verify-Context Consumption Rules

Verify-context must interpret normalized diagnostic outputs consistently.

At minimum, the internal structure must support deterministic mapping from diagnostic status/severity to:
- `can_proceed`
- `requires_repair`
- overall verify status

Avoid rule-specific verify-context branching whenever possible.
Prefer a central mapping strategy.

### 6. Preserve Deterministic Ordering

When multiple diagnostics are produced:
- ordering must be deterministic
- JSON output ordering must remain stable
- the same filesystem and document state must produce the same ordered diagnostics

If explicit sorting is needed, define it centrally.

### 7. Minimize Future Rule Cost

The design should make future rules cheap to add, such as:
- missing canonical context
- malformed context sections
- stale state-document structure
- execution-spec misuse or drift variants

A future rule should ideally be implemented by adding one rule definition rather than editing multiple commands.

### 8. Tests

Add focused coverage proving:

- rules produce normalized outputs consistently
- `execution_spec_drift` works through the new rule model
- doctor aggregation remains deterministic
- verify-context consumes rule outputs consistently
- multiple diagnostics remain stably ordered
- no regressions occur in existing doctor and verify-context behavior
- all tests pass

## Non-Goals
- Do not redesign unrelated CLI commands.
- Do not add third-party extension loading for doctor rules.
- Do not change execution-spec naming or allocation rules.
- Do not invent a second competing diagnostics format.
