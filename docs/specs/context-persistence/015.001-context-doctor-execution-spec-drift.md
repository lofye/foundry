# Execution Spec: 015.001-context-doctor-execution-spec-drift

## Feature
- context-persistence

## Purpose
- Make `foundry context doctor` detect when execution specs exist for a feature but canonical feature context is missing or incomplete.
- Prevent developers and agents from treating execution specs as authoritative in the absence of canonical feature context.
- Align the new diagnostic with the existing deterministic validation style already used by `foundry spec:validate`.

## Scope
- Add a new context diagnostic for execution-spec drift.
- Detect execution-spec presence under `docs/specs/<feature>/` and `docs/specs/<feature>/drafts/`.
- Detect missing canonical context files under `docs/features/`.
- Surface this condition through `context doctor` and `verify context` in a way that matches existing diagnostic behavior and JSON output conventions.
- Keep this focused on detection and reporting only.

## Constraints
- Keep canonical feature context authoritative.
- Keep execution specs non-authoritative.
- Keep output deterministic and machine-readable.
- Reuse existing context-diagnostics and verify-context plumbing where practical.
- Do not auto-create, rewrite, promote, or delete any files.
- Do not infer canonical context from execution specs.
- Do not broaden this into execution-spec naming, allocation, or promotion behavior.

## Requested Changes

### 1. Add a New Diagnostic Condition

Add a new context diagnostic condition named:

`EXECUTION_SPEC_DRIFT`

This condition must be detected when both of the following are true for a feature:

- one or more execution specs exist under:
  - `docs/specs/<feature>/`
  - `docs/specs/<feature>/drafts/`
- one or more required canonical feature context files are missing:
  - `docs/features/<feature>.spec.md`
  - `docs/features/<feature>.md`
  - `docs/features/<feature>.decisions.md`

This diagnostic exists to signal that planning artifacts exist for a feature whose canonical context is missing or incomplete.

### 2. Detection Rules

Detection must be feature-scoped.

For the target feature:

1. Discover whether any execution specs exist in either:
   - the active feature spec directory
   - the draft feature spec directory

2. Compute the set of missing canonical feature context files.

3. Emit `EXECUTION_SPEC_DRIFT` only when:
   - execution specs exist
   - and the missing canonical file set is non-empty

Do not emit the diagnostic when:
- no execution specs exist
- all canonical context files exist

### 3. Diagnostic JSON Shape

The emitted diagnostic must match the deterministic structured style already used by `foundry spec:validate`.

It must include:

- `code`
- `message`
- `file_path`
- `details`

Use this exact code:

`EXECUTION_SPEC_DRIFT`

Use a stable message equivalent to:

`Execution specs exist for this feature, but canonical feature context is missing or incomplete.`

Use `file_path` as the feature-relative specs directory path:

- `docs/specs/<feature>/`

`details` must always be an object, never an array, even when empty.

The `details` object must include:

- `feature`
- `missing_files`
- `execution_specs_found`

It may also include:
- `active_spec_paths`
- `draft_spec_paths`

if those are already easy to provide without making the implementation messy.

### 4. Doctor Output Requirements

`foundry context doctor --feature=<feature> --json` must include this diagnostic in its normal diagnostics list using the standard doctor output shape.

The diagnostic must make clear that:
- execution specs were found
- canonical feature context is incomplete
- execution specs are not authoritative substitutes for canonical feature context

If the doctor command also renders plain text, the plain-text output must be concise, stable, and actionable.

### 5. Verify-Context Integration

`foundry verify context --feature=<feature> --json` must account for this diagnostic in a way that matches the existing context-gating model.

For this implementation:
- treat `EXECUTION_SPEC_DRIFT` as a repair-required context problem
- set proceed/repair signals consistently with the existing verify-context contract
- do not allow meaningful implementation to proceed while this condition exists

The behavior must be conservative and deterministic.

### 6. Repair Guidance

The diagnostic or surrounding command output must guide the developer or agent toward canonical repair steps.

Recommended actions must include guidance equivalent to:

- create or initialize missing canonical context files
- run `foundry context init <feature> --json` when appropriate
- do not rely on execution specs as the source of truth

### 7. Deterministic Ordering

If multiple context diagnostics are present:
- diagnostic ordering must remain deterministic
- repeated runs against the same filesystem state must return the same ordered diagnostics

If sorting is required, define it centrally and keep it stable.

### 8. Tests

Add focused coverage proving:

- no diagnostic is emitted when canonical context exists and execution specs also exist
- no diagnostic is emitted when canonical context is missing but no execution specs exist
- the diagnostic is emitted when execution specs exist and canonical context is missing or incomplete
- active specs trigger the diagnostic
- draft specs trigger the diagnostic
- missing file paths are reported deterministically
- `details` is always an object
- doctor JSON output includes the new diagnostic in the expected structure
- verify-context signals reflect the diagnostic consistently
- all existing related context doctor and verify-context tests still pass

## Non-Goals
- Do not add execution-spec auto-promotion.
- Do not generate canonical context from execution specs.
- Do not change execution-spec naming rules.
- Do not add UI-only behavior here.
- Do not broaden this into a generic docs-health sweep outside feature-scoped context diagnostics.

## Canonical Context
- Canonical feature spec: `docs/features/context-persistence.spec.md`
- Canonical feature state: `docs/features/context-persistence.md`
- Canonical decision ledger: `docs/features/context-persistence.decisions.md`

## Authority Rule
- Canonical feature context remains the only authoritative source of feature intent and state.
- Execution specs remain planning artifacts only.
- When execution specs exist without complete canonical context, the system must diagnose that condition explicitly and require repair before meaningful implementation proceeds.

## Completion Signals
- `context doctor` emits `EXECUTION_SPEC_DRIFT` deterministically when execution specs exist without complete canonical context.
- `verify context` treats this condition as repair-required.
- Missing canonical files are surfaced clearly.
- Diagnostic output uses a stable structured shape.
- `details` is always an object.
- No false positives occur when canonical context is complete.
- All tests pass.

## Post-Execution Expectations
- Developers and agents get an explicit repair-required diagnostic when execution specs exist without canonical feature context.
- The system more strongly enforces context anchoring discipline.
- Execution specs are less likely to be misused as a substitute for canonical feature context.
