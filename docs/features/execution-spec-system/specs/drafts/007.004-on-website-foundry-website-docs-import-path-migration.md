# Execution Spec: 007.004-on-website-foundry-website-docs-import-path-migration

## Purpose

Update the `foundry-website` repository so its docs import, build, navigation, versioning, and publishing logic understands the modular Foundry docs layout introduced by `007-modular-docs-feature-layout.md`.

This spec is for the website repository, not the framework repository.

## Repository

```text
https://github.com/lofye/foundry-website
```

## Depends On

Framework repository specs:

- `007-modular-docs-feature-layout.md`
- `007.001-agent-facing-doc-path-contracts.md`
- `007.002-readmes-stubs-and-public-docs-path-contracts.md`
- `007.003-path-fixtures-tests-and-contract-cleanup.md`

Execute this website spec after the framework repository has completed the modular docs migration.

## Scope

Update all website code that assumes the old framework docs layout, including at minimum:

- docs import scripts
- docs build scripts
- docs sidebar generation
- docs version snapshot generation
- docs sync/update scripts
- GitHub Actions workflows that import framework docs
- tests and fixtures for imported docs
- templates that link to feature docs or execution specs
- any references to `framework/docs/features` or `framework/docs/specs`

## Old Framework Docs Layout

```text
framework/docs/features/<id>-<slug>.md
framework/docs/features/<id>-<slug>.decisions.md
framework/docs/features/<id>-<slug>.spec.md
framework/docs/features/<feature>/specs/<id>-<slug>.md
framework/docs/features/<feature>/specs/drafts/<id>-<slug>.md
```

## New Framework Docs Layout

```text
framework/docs/features/<feature>/<id>-<slug>.md
framework/docs/features/<feature>/<id>-<slug>.decisions.md
framework/docs/features/<feature>/<id>-<slug>.spec.md
framework/docs/features/<feature>/specs/<id>-<slug>.md
framework/docs/features/<feature>/specs/drafts/<id>-<slug>.md
framework/docs/features/<feature>/plans/<id>-<slug>.md
```

## Required Website Behavior

1. Import feature-local docs from `framework/docs/features/<feature>/`.
2. Import feature-local execution specs from `framework/docs/features/<feature>/specs/`.
3. Import implementation plans from `framework/docs/features/<feature>/plans/` when public docs should expose them.
4. Exclude draft specs from public docs unless the website already has an explicit draft-docs mode.
5. Preserve existing versioned docs behavior.
6. Preserve immutable version snapshots.
7. Preserve deterministic sidebar ordering.
8. Preserve the distinction between imported framework docs and authored website docs.

## Implementation Steps

1. Search the website repository for:
   - `docs/features`
   - `docs/specs`
   - `content/docs/imported`
   - `framework/docs`
   - `build-docs`
   - `versions`
2. Identify every import/build script that reads framework docs.
3. Replace old source path assumptions with feature-local traversal.
4. Ensure traversal ignores infrastructure directories unless explicitly supported.
5. Ensure `specs/drafts` are not published by default.
6. Add or update tests covering:
   - feature context docs
   - feature specs
   - decision ledgers
   - execution specs
   - implementation plans
   - draft-spec exclusion
7. Update website docs or README instructions that describe how framework docs are imported.
8. Run the website build and tests.

## Sidebar and Ordering Rules

Docs navigation must be deterministic:

1. Group by feature slug.
2. Within each feature, place feature context/spec/decision docs before `specs/` and `plans/` when those surfaces are included.
3. Sort execution specs by padded hierarchical ID.
4. Sort implementation plans by padded hierarchical ID.
5. Do not depend on filesystem enumeration order.

## Acceptance Criteria

- Website import scripts no longer read from `framework/docs/features/`.
- Website import scripts no longer read execution specs from `framework/docs/features/<feature>/specs/`.
- Imported docs are generated from `framework/docs/features/<feature>/...`.
- Draft specs are excluded from public docs by default.
- Versioned docs snapshots still build.
- Current docs still build.
- Sidebar ordering remains stable across repeated builds.
- Website README/build docs describe the new framework docs layout.

## Verification

Run the website repository's available install, test, and build commands.

At minimum, run the equivalent of:

```bash
grep -R "docs/features" . 2>/dev/null || true
grep -R "docs/specs" . 2>/dev/null || true
```

Any remaining matches must be removed or explicitly marked as historical migration notes.

Then run the website docs build twice and confirm the generated output is deterministic.

## Notes

The website repository must be audited directly during implementation. Do not assume script names or file locations beyond what is discovered in the repository at execution time.
