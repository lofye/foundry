# Execution Spec: 007.004-on-website-foundry-website-docs-import-path-migration

## Feature

- execution-spec-system

## Purpose

Update the `foundry-website` repository so its docs import, build, navigation, versioning, and publishing logic understands Foundry framework's final canonical feature documentation layout.

This is a website-repository spec. It should be handed to Codex while working in the `foundry-website` repository, not while working only inside the framework repository.

## Repository

```text
https://github.com/lofye/foundry-website
```

## Depends On

Framework repository specs:

- `007-modular-docs-feature-layout.md`
- `007.001-agent-facing-doc-path-contracts.md`
- `007.002-readmes-stubs-and-public-docs-path-contracts.md`
- `007.003-path-fixtures-tests-and-contract-cleanup.md`

If `009-correct-feature-docs-layout.md` was already implemented in the framework repository, treat that corrected layout as canonical and do not attempt to import from the intermediate `framework/docs/<feature>/` layout.

## Canonical Framework Docs Layout

Feature context files:

```text
framework/docs/features/<feature>/<feature>.spec.md
framework/docs/features/<feature>/<feature>.md
framework/docs/features/<feature>/<feature>.decisions.md
```

Feature execution specs and drafts:

```text
framework/docs/features/<feature>/specs/<id>-<slug>.md
framework/docs/features/<feature>/specs/drafts/<id>-<slug>.md
```

Feature implementation plans:

```text
framework/docs/features/<feature>/plans/<id>-<slug>.md
```

Feature support files:

```text
framework/docs/features/README.md
framework/docs/features/implementation-log.md
```

## Invalid Active Framework Source Layouts

Website import/build code must not treat these as current source locations:

```text
framework/docs/<feature>/
framework/docs/specs/
framework/docs/features/<id>-<slug>.md
framework/docs/features/<id>-<slug>.spec.md
framework/docs/features/<id>-<slug>.decisions.md
framework/docs/features/<feature>/<id>-<slug>.md
framework/docs/features/<feature>/<id>-<slug>.spec.md
framework/docs/features/<feature>/<id>-<slug>.decisions.md
```

They may appear only as historical migration notes or invalid-path tests.

## Scope

Update all website code that assumes old or intermediate framework docs paths, including at minimum:

- docs import scripts
- docs build scripts
- docs sidebar generation
- docs version snapshot generation
- docs sync/update scripts
- GitHub Actions workflows that import framework docs
- tests and fixtures for imported docs
- templates that link to feature docs, execution specs, draft specs, plans, or the implementation log
- README/build docs that describe framework-doc imports
- any references to `framework/docs/features`, `framework/docs/specs`, or `framework/docs/<feature>`

## Required Website Behavior

1. Import feature context docs from `framework/docs/features/<feature>/`.
2. Import feature-local execution specs from `framework/docs/features/<feature>/specs/`.
3. Exclude `framework/docs/features/<feature>/specs/drafts/` from public docs unless the website already has an explicit draft-docs mode.
4. Import implementation plans from `framework/docs/features/<feature>/plans/` only when public docs intentionally expose implementation planning artifacts.
5. Import or link `framework/docs/features/implementation-log.md` only where the website intentionally exposes completed execution history.
6. Preserve the distinction between imported framework docs and authored website docs.
7. Preserve existing versioned docs behavior.
8. Preserve immutable version snapshots.
9. Preserve deterministic sidebar ordering.
10. Do not publish draft specs by default.

## Sidebar and Ordering Rules

Docs navigation must be deterministic:

1. Group imported feature docs by feature slug.
2. Within each feature, place feature context docs before execution specs and plans when those surfaces are included.
3. Order feature context docs consistently:
   - `<feature>.spec.md`
   - `<feature>.md`
   - `<feature>.decisions.md`
4. Sort execution specs by padded hierarchical ID.
5. Sort implementation plans by padded hierarchical ID.
6. Do not depend on filesystem enumeration order.

## Implementation Steps

1. Search the website repository for old and intermediate path references:
   - `docs/features`
   - `docs/specs`
   - `docs/<feature>`
   - `framework/docs`
   - `content/docs/imported`
   - `build-docs`
   - `versions`
2. Identify import/build scripts that read framework docs.
3. Replace source traversal with canonical `framework/docs/features/<feature>/...` traversal.
4. Ensure traversal skips infrastructure files and directories unless explicitly supported:
   - `framework/docs/features/README.md`
   - `framework/docs/features/implementation-log.md`
   - `specs/drafts/`
5. Add or update tests covering:
   - feature spec import
   - feature state import
   - decision ledger import
   - execution spec import
   - draft spec exclusion
   - implementation plan import or intentional exclusion
   - deterministic sidebar ordering
6. Update website README/build docs with the canonical framework layout.
7. Run the website build and test suite.

## Acceptance Criteria

- Website import scripts read feature docs from `framework/docs/features/<feature>/...`.
- Website import scripts do not read from `framework/docs/<feature>/...`.
- Website import scripts do not read from `framework/docs/specs/...`.
- Website import scripts do not treat feature context files as `<id>-<slug>.*.md`.
- Draft specs are excluded from public docs by default.
- Plans are included only if the website intentionally exposes them.
- Current docs build succeeds.
- Versioned docs snapshots still build.
- Sidebar ordering is deterministic across repeated builds.
- Website README/build docs describe the canonical framework layout.

## Verification

Run the website repository's available install, test, lint, and build commands.

At minimum:

```bash
grep -R "framework/docs/<feature>" . 2>/dev/null || true
grep -R "framework/docs/specs" . 2>/dev/null || true
grep -R "framework/docs/features/<id>-<slug>" . 2>/dev/null || true
grep -R "framework/docs/features/<feature>/<id>-<slug>" . 2>/dev/null || true
```

Any remaining matches must be removed, corrected, or explicitly labeled as historical or invalid-path test coverage.

Run the website docs build twice and confirm generated output is deterministic.

## Implementation Log

If the website repository has its own implementation-log convention, follow it. Do not append a framework repository implementation-log entry from the website repository unless the framework workflow explicitly requires it.
